## 2. DB物理設計

### users

会員・管理者を保持する。メールは小文字正規化し一意とする。パスワードはハッシュだけを保存する。

主要項目: `id`, `email`, `password_hash`, `display_name`, `locale`, `role`, `status`, `email_verified_at`, `trial_granted_at`, `terms_version`, `terms_accepted_at`, `created_at`, `updated_at`, `deleted_at`。

### products

Stripe商品と付与秒数を対応付ける。価格・秒数はブラウザから受け取らずDBを正とする。

### wallets / credit_ledger

`wallets` は表示高速化のため無料・有料残高を保持する。すべての増減は `credit_ledger` に追記し、同じDBトランザクション内で残高を更新する。`idempotency_key` は一意とする。

種別: `trial_grant`, `purchase`, `usage`, `expiry`, `refund`, `chargeback`, `admin_adjustment`, `reservation`, `release`。

### payments / stripe_events

Checkout Session、Payment Intent、金額、通貨、状態を保持する。Webhookは最初に`stripe_events`へ一意登録し、重複時は成功応答のみ返す。

### translation_sessions / usage_events

音声本文を保存せず、開始・終了、言語、利用秒数、状態、技術エラー分類だけを保持する。

## 3. API仕様

すべてJSON。成功は `{ "ok": true, ... }`、失敗は `{ "ok": false, "error": { "code": "...", "message": "..." } }`。

| POST | `/api/auth/login.php` | 不要 | セッションログイン |

## 4. 無料付与処理

1. メール確認トークンをハッシュ照合し、有効期限・未使用を確認する。
2. `users.email_verified_at` を設定する。
3. `users.trial_granted_at IS NULL` の場合だけ付与する。
4. `credit_ledger`へ`trial_grant` 900秒を一意キー付きで追加する。
5. `wallets.trial_seconds`へ900秒を加算する。
6. トークン使用済みとする。
7. 全処理を一つのトランザクションで確定する。

## 5. 利用時間確定

- Gatewayは最大30秒ずつ利用枠を取得する。
- 無料残高から先に予約し、不足分を有料残高から予約する。
- セッション切断時は実利用秒へ精算し、未使用予約分を戻す。
- Gatewayからの各要求はセッションIDと連番を一意キーにする。
- 一定時間応答がない予約は回収ジョブで解放する。
- ブラウザからの秒数申告は残高確定に使用しない。

## 6. Stripe Webhook

- raw request bodyを使って`Stripe-Signature`を検証する。
- 許容時刻差は300秒とする。
- `checkout.session.completed`かつ`payment_status=paid`で付与する。
- Checkout Sessionのmetadataには内部`user_id`と`product_id`だけを入れる。
- 金額・通貨・付与秒数はDB商品と照合する。
- 返金とチャージバックは別イベントで残高調整・管理者通知する。

## 7. 認証・セッション

- Cookie: `HttpOnly; Secure; SameSite=Lax`
- ログイン成功時にセッションIDを再生成する。
- セッションはサーバー側で30分無操作失効、最大12時間とする。
- ログイン失敗はIP・メール単位で制限する。
- 認証エラーはメール存在有無が分からない同一文言とする。
- 管理者はTOTP二要素認証を必須とする。

## 8. エラーコード

`AUTH_REQUIRED`, `EMAIL_NOT_VERIFIED`, `ACCOUNT_DISABLED`, `CSRF_INVALID`, `VALIDATION_ERROR`, `BALANCE_EMPTY`, `PAYMENT_NOT_CONFIRMED`, `LANGUAGE_PAIR_INVALID`, `MIC_PERMISSION_DENIED`, `REALTIME_UNAVAILABLE`, `RATE_LIMITED`, `INTERNAL_ERROR`。

画面表示文はUIの翻訳辞書でローカライズし、APIは安定したコードを返す。

## 9. ログ

記録する: request ID、会員ID、管理者ID、処理種別、HTTP状態、処理時間、決済イベントID、利用秒数、エラー分類。  
記録しない: パスワード、トークン、Cookie、APIキー、カード情報、音声、翻訳本文。

## 12. AI会議準備ブリーフ詳細（Build Week追加機能）

### 12.1 API入力

`POST /api/preparation/generate.php`

```json
{
  "source_language": "ja",
  "target_language": "en",
  "situation": "海外顧客との製品説明会",
  "key_terms": "ShalomWorks, Realtime API, 15分無料"
}
```

- `source_language`, `target_language`: 許可リスト内かつ異なる言語
- `situation`: 1〜1,200文字
- `key_terms`: 0〜800文字
- 合計2,000文字以内
- Cookie認証とCSRFトークン必須

### 12.2 GPT-5.6呼び出し

- Endpoint: `POST https://api.openai.com/v1/responses`
- Model: `gpt-5.6`
- `store: false`
- `reasoning.effort: low`
- `safety_identifier`: 会員IDのSHA-256ハッシュ
- Structured Outputsを使用し、次のJSON構造に固定する。

```json
{
  "summary": "string",
  "terms": [
    { "source": "string", "target": "string", "note": "string" }
  ],
  "risks": ["string"],
  "speaking_tips": ["string"],
  "checklist": ["string"]
}
```

配列上限は`terms` 8件、その他各5件とする。API応答後もPHP側で型、件数、一項目の文字数を検証する。

### 12.3 回数制限

`ai_preparation_usage`に`user_id`, `usage_date`, `request_count`, `updated_at`を保持し、`(user_id, usage_date)`を一意とする。トランザクション内で当日件数を確認し、5回以上は`RATE_LIMITED`を返す。OpenAI API呼び出し失敗時は件数を戻す。

### 12.4 非保存・ログ

保存しない: `situation`, `key_terms`, GPT-5.6の出力全文、将来の音声・通訳本文。
記録する: 会員ID、日付、回数、HTTP状態、処理時間、OpenAI request ID、入力・出力トークン数、エラー分類。

### 12.5 エラー

- `VALIDATION_ERROR`: 入力不正、同一言語、文字数超過
- `RATE_LIMITED`: 1日5回を超過
- `AI_PREPARATION_UNAVAILABLE`: OpenAI API障害、タイムアウト、応答構造不正
- `AUTH_REQUIRED`, `CSRF_INVALID`: 既存共通処理を使用

### 12.6 表示上の注意

「AIが作成した準備情報です。重要な名称・数字・専門用語は利用者が確認してください」と3言語で表示する。生成結果はRealtime Translateの精度保証ではない。
